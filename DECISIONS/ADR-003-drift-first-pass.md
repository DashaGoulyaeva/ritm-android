# ADR-003: Текущий архитектурный drift и допустимые отклонения (первый проход)

**Статус:** Принято  
**Дата:** 2026-04-20

## Контекст

Репозиторий содержит рабочий Foundation и частично реализованный модуль `feature-water`,
при этом `TASKS.md` ещё не отражает фактический прогресс по водному slice.
Также пока отсутствуют Git remote, доступный Android SDK и проверенные plugin-интеграции.

## Решение

1. На первом проходе считаем допустимым временный drift между roadmap и кодом,
   если после каждого завершённого slice синхронизируются `TASKS.md` и `docs/devlog.md`.
2. `feature-water` развиваем без изменения публичных контрактов:
   `WaterRepository`, `WaterGoalRepository`, `WaterUiState`, use cases.
3. Навигацию верхнего уровня не ломаем: `TopLevelDestination` остаётся прежним,
   а water quick-log подключается только внутри Today-route.
4. Инфраструктурные задачи preflight (`git/sdk/plugins`) фиксируются как `DONE` или `BLOCKED`
   с явным обоснованием в devlog.

## Последствия

- (+) Можно продолжать MVP-разработку без стопа на окружении, сохраняя прозрачность статуса.
- (+) Риски архитектурного расползания контролируются через явную синхронизацию docs.
- (-) До настройки SDK нельзя полноценно верифицировать сборку и quality gates.
- (-) До подключения plugin-интеграций часть GitHub/Figma-потока остаётся локальной.
