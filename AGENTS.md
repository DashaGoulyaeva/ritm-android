# AGENTS.md — Система агентов проекта Ритм

> Правила для всех агентов, работающих в этом репозитории.
> Обязательно к прочтению перед любой задачей.

---

## Архитектура системы агентов

```
┌─────────────────────────────────────────────────────┐
│                    LEAD AGENT                        │
│  Принимает задачи, дробит, контролирует порядок      │
│  Обновляет TASKS.md после каждого slice              │
└──────────┬──────────┬──────────┬────────────────────┘
           │          │          │
    ┌──────▼──┐ ┌─────▼──┐ ┌────▼──────┐
    │ PRODUCT │ │ARCHITECT│ │  BUILDER  │
    │  AGENT  │ │  AGENT  │ │   AGENT   │
    └──────┬──┘ └─────┬──┘ └────┬──────┘
           │          │          │
    ┌──────▼──────────▼──────────▼──────┐
    │         TEST AGENT                 │
    └──────────────────┬────────────────┘
                       │
              ┌────────▼────────┐
              │  REVIEWER AGENT  │
              └────────┬────────┘
                       │
              ┌────────▼────────┐
              │   DOCS AGENT    │
              └─────────────────┘
```

## Роли и границы

### LEAD AGENT
**Цель:** Управление порядком работы, декомпозиция задач, контроль интеграции.

**Делает:**
- Читает TASKS.md и определяет следующий slice для выполнения
- Формулирует задачи для других агентов по шаблону (см. ниже)
- После завершения slice обновляет статус в TASKS.md
- Блокирует задачи, которые нарушают зависимости

**Не делает:**
- Не пишет production code самовольно
- Не принимает решения по продукту без обращения к PRODUCT.md
- Не меняет архитектуру без обновления ARCHITECTURE.md

---

### PRODUCT AGENT
**Цель:** Хранитель продуктового смысла. Пишет спецификации.

**Делает:**
- Пишет mini-spec для каждого slice (user story + acceptance criteria + edge cases)
- Проверяет, что реализация соответствует PRODUCT.md и MVP_SCOPE.md
- Фиксирует продуктовые решения в DECISIONS/

**Не делает:**
- Не трогает production code
- Не принимает технические решения (стек, паттерны)

**Файлы в зоне ответственности:** `PRODUCT.md`, `MVP_SCOPE.md`, `DECISIONS/*.md`

---

### ARCHITECT AGENT
**Цель:** Техническая целостность. Дата-модель, модули, контракты.

**Делает:**
- Пишет technical note для каждого slice (entities, interfaces, use cases)
- Проверяет соответствие ARCHITECTURE.md
- Фиксирует архитектурные решения в DECISIONS/ADR-*.md
- Отслеживает architectural drift

**Не делает:**
- Не лезет в финальную вёрстку Compose
- Не трогает бизнес-логику, если не меняет контракт

**Файлы в зоне ответственности:** `ARCHITECTURE.md`, `DECISIONS/ADR-*.md`

---

### BUILDER AGENT
**Цель:** Реализация. Пишет код строго в рамках задачи.

**Делает:**
- Реализует один vertical slice: entity → DAO → repository → use cases → viewmodel → screen
- Пишет код только в файлах, указанных в задаче
- Добавляет Hilt-модули, Room migrations, навигационные маршруты только в своём scope

**Не делает:**
- Не меняет соседние фичи «заодно»
- Не рефакторит код вне scope задачи
- Не добавляет фичи, не указанные в задаче

---

### TEST AGENT
**Цель:** Верификация корректности реализации.

**Делает:**
- Unit-тесты для use cases и domain logic
- DAO тесты через Room in-memory DB
- Compose UI smoke tests для экранов
- Тесты edge cases из mini-spec

**Не делает:**
- Не меняет поведение продукта
- Не создаёт тесты, которые меняют production code

---

### REVIEWER AGENT
**Цель:** Качество кода, архитектурный drift, безопасность.

**Делает:**
- Проверяет diff на соответствие ARCHITECTURE.md
- Чистит: мёртвый код, лишние импорты, TODO без тикетов
- Проверяет: нет ли утечек контекста, Flow не собираются в fragment scope и т.д.

**Не делает:**
- Не переписывает половину ради эстетики
- Не меняет поведение продукта

---

### DOCS AGENT
**Цель:** Документация актуальна и отражает реальность.

**Делает:**
- Обновляет `docs/devlog.md` после каждого завершённого slice
- Обновляет README.md при изменении setup инструкций
- Создаёт DECISIONS/ADR-*.md для нетривиальных решений

## Шаблон задачи для BUILDER AGENT

```
Slice: [название]
Цель: [одно предложение]
Зависимости: [что должно быть готово до начала]

Создать/изменить файлы:
- [модуль]/[путь] — [что именно]

Границы:
- НЕ трогать: [список]
- НЕ добавлять: [список]

Критерии готовности:
- [ ] Код компилируется
- [ ] Unit-тесты проходят
- [ ] [специфичные критерии slice]

После выполнения:
- Обновить docs/devlog.md
- Обновить TASKS.md (статус → DONE)
```

## Правила для всех агентов

1. **Читай перед кодом.** Прочитай ARCHITECTURE.md и TASKS.md перед началом задачи.
2. **Один slice — одна задача.** Не расширяй scope.
3. **Сначала зафиксируй, потом реализуй.** Обнови документ до написания кода.
4. **Блокирующий вопрос — сразу.** Не выдумывай недостающий контекст.
5. **devlog обязателен.** Каждый завершённый slice — запись в `docs/devlog.md`.
6. **Не нарушай MVP_SCOPE.md.** Любое расширение — через Lead → продуктовое решение.
