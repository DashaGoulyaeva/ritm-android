# Rebuild Plan

Статус: рабочий план реворка.

## Принцип

Сначала смысл, потом структура, потом код. Старое MVP хранится как архив и источник уроков, но не диктует новую архитектуру продукта.

## Этапы

1. Заморозить MVP v1 тегом `mvp-v1-baseline`.
2. Работать в ветке `codex/product-reset-v2`.
3. Заполнить `UX_REVIEW_V1.md` по фактическому опыту использования.
4. Сформулировать `PRODUCT_V2.md`.
5. Утвердить `MVP_SCOPE_V2.md`.
6. Разметить текущие модули в `MODULE_DECISIONS.md` как `KEEP`, `REWORK`, `DROP` или `UNKNOWN`.
7. Выбрать техническую стратегию: мягкий rebuild или новый app shell внутри репозитория.
8. Только после этого нарезать первые задачи на реализацию.

## Правила веток

- `main` — публичная стабильная линия.
- `v1.0-dev` — сохранённая линия MVP v1 / dev-состояние до реворка.
- `codex/product-reset-v2` — рабочая ветка переосмысления.

## Запрещено на этапе реворка

- удалять v1-файлы без отдельного решения;
- переписывать код до утверждения продукта v2;
- тащить старые экраны в v2 только потому, что они уже готовы;
- смешивать документы v1 и v2 без явной пометки статуса.
