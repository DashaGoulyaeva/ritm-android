# CONTRIBUTING

Спасибо за интерес к `ritm-android`!  
Ниже короткий гайд, чтобы изменения вливались быстро и без лишнего шума.

## Перед началом

1. Прочитайте:
   - [PRODUCT.md](./PRODUCT.md)
   - [MVP_SCOPE.md](./MVP_SCOPE.md)
   - [ARCHITECTURE.md](./ARCHITECTURE.md)
   - [TASKS.md](./TASKS.md)
2. Проверьте, что задача укладывается в MVP (без облака/AI/медицинских обещаний).
3. Если задачи нет в `TASKS.md`, сначала откройте issue/обсуждение.

## Правила вкладов

- Один PR = один логический slice.
- Без незапрошенных рефакторингов соседних модулей.
- Не меняйте поведение продукта вне scope задачи.
- Сохраняйте модульные границы (`feature-*` не зависят друг от друга напрямую).

## Локальная проверка перед PR

```bash
./gradlew :app:assembleDebug
./gradlew qualityCheck
```

Для Windows PowerShell: `.\gradlew.bat`.

## Оформление PR

- Понятный заголовок и описание: что сделано и зачем.
- Ссылка на задачу из `TASKS.md`/issue.
- Кратко указать, какие риски и что проверяли.
- Если менялись публичные правила/процессы — обновить соответствующий `.md`.

## Что точно не принимается в MVP-ветку

- фичи из post-MVP без отдельного решения;
- интеграция облака, аккаунтов, AI;
- медицинские обещания/диагностические формулировки;
- смешение нескольких крупных задач в одном PR.
